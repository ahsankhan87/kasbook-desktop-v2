# Stock Check & Adjustment Module — Database Deployment Guide

## Script execution order

Run each script **once** in this exact order against your SQL Server database.

| Order | File | Purpose |
|-------|------|---------|
| 1 | `StockAdjustmentSchema.sql` | Creates all tables, constraints, indexes, trigger, views, and `sp_GenerateAdjNo` |
| 2 | `StockAdjustmentProcedures.sql` | Creates/replaces all session lifecycle and report stored procedures |

The older `StockAdjustmentAudit.sql` is **superseded** by the two files above and should **not** be re-run.

## Tables created by `StockAdjustmentSchema.sql`

### `stk_adj_sessions`
Adjustment session header. One row per stock-check/adjustment event.

| Column | Type | Notes |
|--------|------|-------|
| `adj_id` | INT IDENTITY PK | Auto-generated |
| `adj_no` | VARCHAR(20) UNIQUE | Generated by `sp_GenerateAdjNo` (e.g. `PHY-2025-0001`) |
| `adj_date` | DATE | Defaults to today |
| `adj_type` | VARCHAR(30) | One of: Physical Count, Damage Write-Off, Found/Excess, Price Update, Location Transfer, Opening Stock |
| `warehouse_id` | INT FK → warehouses | |
| `status` | VARCHAR(10) | `Draft` → `Posted` → `Reversed` |
| `total_lines` / `qty_increases` / etc. | INT | Counters updated by `sp_PostStockAdjustment` |
| `created_by`, `posted_by`, `reversed_by` | INT FK → users | Lifecycle actors |
| `reversal_reason` | NVARCHAR(500) | Required when reversing |

### `stk_adj_lines`
One row per product per session.

| Column | Type | Notes |
|--------|------|-------|
| `line_id` | INT IDENTITY PK | |
| `adj_id` | INT FK → stk_adj_sessions | |
| `product_id` | INT FK → mst_products | UNIQUE with adj_id |
| `system_qty` / `physical_qty` | DECIMAL(18,3) | |
| `qty_difference` | DECIMAL(18,3) **PERSISTED** | `physical_qty - system_qty` (computed) |
| `current_sale_price` / `new_sale_price` | DECIMAL(18,2) | |
| `price_difference` | DECIMAL(18,2) **PERSISTED** | `new_sale_price - current_sale_price` (computed) |
| `current_location` / `new_location` | NVARCHAR(30) | Format: `AISLE-SHELF-BIN` |
| `reason` | VARCHAR(30) | Physical Count, Damage, Found, etc. |
| `is_verified` | BIT | 0 = unverified, 1 = verified by supervisor |

### `stk_adj_audit`
Immutable append-only audit log. Protected by `TR_stk_adj_audit_immutable` trigger.

| Column | Type | Notes |
|--------|------|-------|
| `audit_id` | BIGINT IDENTITY PK | |
| `change_type` | VARCHAR(20) | `QTY_CHANGE`, `PRICE_CHANGE`, `LOCATION_CHANGE` |
| `old_value` / `new_value` | NVARCHAR(100) | String representation of the values |
| `changed_by` | INT FK → users | |
| `adj_no` | VARCHAR(20) | Denormalised for fast report queries |

> **Never** run UPDATE or DELETE on this table — the trigger will raise an error and roll back.

### `mst_product_locations`
Multi-bin location stock. One row per product+warehouse+bin combination.

| Column | Type | Notes |
|--------|------|-------|
| `loc_id` | INT IDENTITY PK | |
| `product_id` / `warehouse_id` | INT FK | |
| `aisle_code` / `shelf_code` / `bin_code` | VARCHAR(10) | Unique triplet per product/warehouse |
| `location_code` | VARCHAR(32) **PERSISTED** | `aisle-shelf-bin` (computed) |
| `qty_at_location` | DECIMAL(18,3) | Must be ≥ 0 |
| `is_primary` | BIT | 1 = primary bin for this product/warehouse |

## Stored procedures created by `StockAdjustmentProcedures.sql`

### Session lifecycle
| Procedure | Called from | Description |
|-----------|-------------|-------------|
| `sp_StockAdjustment_CreateSession` | `StockAdjustmentDLL.CreateSession` | Generates adj_no, inserts Draft session, returns adj_id + adj_no |
| `sp_StockAdjustment_UpsertLine` | `StockAdjustmentDLL.SaveDraft` | Insert or update one adjustment line (Draft only) |
| `sp_StockAdjustment_DeleteLine` | (future) | Remove a line from a Draft session |
| `sp_StockAdjustment_GetSessionSummary` | `StockAdjustmentDLL.GetSessionSummary` | Returns aggregate counts/values for one session |
| `sp_PostStockAdjustment` | `StockAdjustmentDLL.PostAdjustment` | Applies all changes, writes audit rows, marks Posted |
| `sp_StockAdjustment_Reverse` | `StockAdjustmentDLL.ReverseAdjustment` | Rolls back changes, writes reversal audit rows, marks Reversed |

### Reporting
| Procedure | Called from | Description |
|-----------|-------------|-------------|
| `sp_GetAdjustmentHistory` | `StockAdjustmentDLL.GetAdjustmentHistory` | Chronological audit feed for one product |
| `sp_GetAdjustmentSessions` | `StockAdjustmentDLL.GetAdjustmentSessions` | Session list with counters for the voucher report |
| `sp_StockVarianceReport` | `StockAdjustmentDLL.GetStockVarianceReport` | Products ranked by adjustment frequency/value |
| `sp_PriceChangeReport` | `StockAdjustmentDLL.GetPriceChangeReport` | All price changes in a date range |

### Number generation
`sp_GenerateAdjNo @AdjType VARCHAR(30), @AdjNo VARCHAR(20) OUTPUT`

Returns the next sequential adjustment number for the given type and calendar year:

| adj_type | Prefix | Example |
|----------|--------|---------|
| Physical Count | PHY | `PHY-2025-0001` |
| Damage Write-Off | DAM | `DAM-2025-0001` |
| Found/Excess | FND | `FND-2025-0001` |
| Price Update | PRC | `PRC-2025-0001` |
| Location Transfer | LOC | `LOC-2025-0001` |
| Opening Stock | OPN | `OPN-2025-0001` |
| (any other) | ADJ | `ADJ-2025-0001` |

## Notes on `sp_PostStockAdjustment` stock balance update

The procedure updates `mst_product_stock.qty_on_hand`. If your stock balance table
has a different name or structure, locate this section in `StockAdjustmentProcedures.sql`
and adjust the UPDATE target accordingly before running the script:

```sql
-- ---- Apply stock qty changes --------------------------------
UPDATE ps SET ps.qty_on_hand = ps.qty_on_hand + l.qty_difference
FROM mst_product_stock ps ...
```
